# 📌 1. 本仕様書の目的

本仕様書は、研究室内で運用される複数の Web アプリケーションを、専門知識の少ない利用者でも安全かつ直感的に追加・公開・再起動・監視できるようにするための、統合配信基盤「Lab-Core v3」の詳細仕様を定義する。

本システムは、従来の属人的な Docker / Proxy / DNS 運用を廃止し、以下を実現することを目的とする。

- 研究室メンバーが怖がらずに操作できる UI
- アプリ開発とインフラ運用の責務分離
- ログ確認と再起動を中心とした単純な障害復旧導線
- OruCa を含む主要アプリの安定運用
- 卒業・引き継ぎ後も継続可能な運用手順の確立

---

# 🎯 2. システムの基本方針

## 2.1 ミッション
Lab-Core v3 は、研究室内の Web アプリを「誰でも保守運用できる」「壊しにくい」「壊れても復元しやすい」状態で提供するための内部向け配信基盤である。

## 2.2 解決すべき課題
本システムは、以下の課題を解決対象とする。

- 設定ファイル管理が難しく、属人化している
- ログ確認や障害対応がしづらい
- 他の研究室メンバーが、再起動や更新を怖がって実施できない
- アプリ側の構成がインフラ都合に引きずられやすい
- PC 再起動や移行時の復旧手順が不明瞭

## 2.3 非目的
本システムは、以下を目的としない。

- 汎用的な Kubernetes 代替基盤の実現
- 企業向け高権限統制・監査・SSO の実装
- 研究室外公開前提のゼロトラスト設計
- 非 HTTP サービス全般の統合管理
- 各アプリ固有の業務機能やバックアップ機能の代替

# 🧠 4. 設計原則

## 4.1 Fearless Operation
利用者が「触ったら壊れるかもしれない」と感じないことを最優先する。

## 4.2 Minimal Infrastructure Imposition
アプリにインフラ都合を強く押しつけない。  
ただし、基盤に載せるための最小契約は定義する。

## 4.3 Simple Recovery Path
障害時の操作順序を固定する。

1. 再起動
2. ログ確認
3. 再ビルド
4. データ保持の有無を選択した再構築

## 4.4 Internal Platform First
研究室 LAN 内部向けに最適化し、外部公開向けの複雑性は持ち込まない。

## 4.5 OruCa First
初期版の成功基準は OruCa を安定的に動作させられることとする。

## 4.6 Responsibility Separation
- アプリの設定・機能・データの意味づけはアプリ側
- 配備・公開・観察・再起動は基盤側

# 🏗️ 5. システムの位置づけ

Lab-Core v3 は、研究室向け軽量 PaaS 的性質を持つが、完全な PaaS ではない。  
正確には、以下の 4 機能を担う内部基盤である。

- Application Deployment
- Reverse Proxy Orchestration
- Internal DNS Routing
- Operational Observation

対象機能:

# 🧰 7. 対応アプリの受け入れ仕様

## 7.1 アプリ受け入れの基本原則
Lab-Core v3 は任意の Docker 構成を無制限に受け入れない。  
運用容易性と引き継ぎ性を優先し、受け入れ可能なアプリの条件を定義する。

## 7.2 アプリ受け入れ条件
アプリは、以下を満たす必要がある。

- Git リポジトリとして取得可能であること
- 単体で起動可能な配備定義を持つこと
- 配信対象となる HTTP コンテナが 1 つ以上特定できること
- 公開対象ポートが明示できること
- 永続データの保存先がコンテナ外に分離できること
- 再ビルド手順が自動化可能であること
- 開発用構成と配信用構成が、可能な限り単純であること

## 7.3 初期版で推奨するアプリ構造
- 1 アプリ = 1 配備定義
- 1 アプリ = 1 公開対象 HTTP サービス
- 補助コンテナは同一アプリに属する内部構成として扱う
- 開発用サーバーと本番配信サーバーの二重管理を避ける
- profile に強く依存しない

## 7.4 特殊要件の扱い
NFC リーダーや USB デバイスなど、ホスト接続機器を必要とするアプリは、オプション機能として「Device Requirement」を宣言可能とする。

例:
- OruCa の NFC リーダー
- シリアル機器
- 特定ホストデバイス

# 🧭 8. アーキテクチャ概要

## 8.1 コンポーネント構成
1. Dashboard
2. Proxy Controller
3. Reverse Proxy
4. Local DNS
5. Git / Build Engine
6. Docker Runtime Controller
7. Log Collector
8. State Store

## 8.2 論理構成
- Dashboard: ユーザー操作、状態表示、アプリ登録
- State Store: アプリ定義、配信定義、状態、イベント記録
- Runtime Controller: Docker に対する起動・停止・再ビルド命令
- Proxy Controller: 配信ルートを reverse proxy に反映
- DNS: `*.fukaya-sus.lab` を内部 IP に解決
- Log Collector: 各コンテナログ、共通基盤ログを統一収集

## 8.3 基本構成技術
| 分類 | 推奨技術 |
| :--- | :--- |
| Runtime | Node.js 20+ |
| Language | TypeScript |
| Backend | Hono |
| Frontend | React 系 SPA |
| DB | SQLite |
| Reverse Proxy | Caddy |
| DNS | dnsmasq または同等の軽量内部 DNS |
| Container Control | Docker Engine API |
| Git Integration | simple-git または同等 |

# 🚀 12. 主要機能仕様

## 12.1 アプリ登録
利用者は以下を入力する。

- GitHub URL
- 公開対象コンテナ名
- 公開ポート
- 公開サブドメイン
- Headless / Standard の別
- 必要に応じて特殊デバイス要件

システムは以下を自動実施する。

- リポジトリ clone
- 配備定義の読み込み
- コンテナビルド
- 起動
- `hostname -> container:port` のルーティング設定
- 公開確認
- 一覧への反映

## 12.2 アプリ更新
更新方法:
- 手動更新
- 定期確認による更新通知

更新仕様:
- 現在コミットを保持
- リモート最新コミットを取得
- 差分の有無を表示
- 更新時は 1 つ前のコミットを保持
- 失敗時は直前コミットへロールバック可能

## 12.3 アプリ再起動
機能要件:
- アプリ単位の再起動
- 必要に応じて属する全コンテナを再起動
- 再起動中は UI 上で状態を明示
- 再起動の成功・失敗をイベント表示

## 12.4 アプリ削除
削除モード:
- 構成のみ削除
- 構成 + ソース削除
- 構成 + ソース + データ削除

要求事項:
- 破壊的操作は二重確認
- データ削除時は赤色系の危険表示
- 説明文を日本語で明示
- 誤削除防止の確認文入力を要求可能

## 12.5 アプリ再ビルド
再ビルドモード:
- Rebuild with Data
- Rebuild without Data

要求事項:
- 再ビルド中の状態可視化
- ログ表示
- データ削除時の明確な警告
- 失敗時の詳細ログ保持

## 12.6 ログ監視
ログは本システムの最重要監視対象である。

対象ログ:
- アプリ各コンテナ標準出力
- Reverse Proxy ログ
- DNS ログ
- Dashboard システムログ

表示要件:
- アプリ画面から属するコンテナログを一覧閲覧可能
- コンテナごとに切替可能
- 重要ログを視覚的に強調
- 日本語 UI で用語を簡単にする
- オートスクロールのオン・オフ切替
- エラーログ発生時に一覧画面へバッジ表示

## 12.7 メトリクス
初期版では最小限とする。

許容範囲:
- 稼働状態
- 最終再起動時刻
- コンテナ数
- 稼働時間

CPU / メモリの詳細監視は後続フェーズとする。

## 12.8 自動再起動
要求事項:
- コンテナ異常終了時に自動再起動
- 再起動回数の表示
- 繰り返し異常時は Degraded として表示
- 利用者に「いま不安定である」ことをやさしい言葉で提示

# 🌐 13. DNS / Proxy 仕様

## 13.1 ドメイン体系
- ルートドメイン: `fukaya-sus.lab`
- アプリ公開: `{subdomain}.fukaya-sus.lab`
- API 特化公開も同様にサブドメイン形式を用いる
- 固定 IP:
  - Main: `192.168.11.224`
  - SSH: `192.168.11.225`

## 13.2 DNS 要件
- `*.fukaya-sus.lab` を `192.168.11.224` に解決可能であること
- 例外ホストは明示設定できること
- `ssh.fukaya-sus.lab` は `192.168.11.225` を向けられること
- DNS ログの基本確認が UI から可能であること

## 13.3 Proxy 要件
- ホスト名ベースのルーティング
- Dashboard 主導で設定管理
- 対象アプリ削除時に route を除去
- 再起動後に route を再同期可能
- route の重複生成を防止
- Proxy ログを共通形式で表示

## 13.4 Headless Service
「ヘッドレス」とは UI を持たないが HTTP エンドポイントを持つサービスを指す。  
UI 上は通常アプリと同様に扱うが、表示ラベル上で区別できるようにする。

# 📈 21. 非機能要件

## 21.1 可用性
- OruCa を安定運用できること
- ホスト再起動後に基盤が自動起動できること
- アプリの再起動が容易であること

## 21.2 保守性
- ディレクトリ構成が単純であること
- 手順書で再構築可能であること
- DB と設定の関係が明確であること

## 21.3 引き継ぎ性
- 運用手順が UI 主体で完結すること
- 深い Docker 知識なしで日常操作が可能であること

## 21.4 性能
- 小規模研究室利用を前提とする
- 同時大量アクセスの最適化は対象外
- ログ表示は実用範囲でリアルタイム性を確保

## 21.5 セキュリティ
本仕様では、高度な権限分離や外部公開向けセキュリティ設計を主対象としない。  
ただし、内部システムとして最低限の誤操作防止は UI / 操作設計により担保する。

# 📝 27. 最終定義

Lab-Core v3 は、研究室内の Web アプリ運用を、詳しい人だけの作業から、普通の研究室メンバーでも落ち着いて扱える日常作業へ変えるための内部配信基盤である。

その本質は、機能の多さではなく次の 4 点にある。

- 怖くないこと
- 壊しにくいこと
- 戻しやすいこと
- 引き継げること
