# 使い方ガイド

最終更新: 2026-03-19

## 1. このガイドの対象

このドキュメントは、現在の Lab-Core v3 ダッシュボード実装を前提にした操作マニュアルです。

対象:
- `core/dashboard` の運用 UI
- `core/backend` が提供する運用 API
- GitHub からのアプリ登録、配備、観測、更新、削除の一連の流れ

このガイドは「理想仕様」ではなく、2026-03-19 時点で実際に動作している UI と挙動を説明します。

## 2. 起動とアクセス

### 2.1 初回セットアップ
1. リポジトリルートで `yarn install`
2. 設定ウィザードを実行: `yarn config:init`

### 2.2 開発時の起動
1. backend を起動: `yarn dev:backend`
2. dashboard を起動: `yarn dev:dashboard`
3. ブラウザで `http://localhost:5173` を開く

### 2.3 実運用確認時の前提
- `execute` モードでは Docker が利用可能であること
- GitHub から対象リポジトリにアクセスできること
- 生成される公開設定を反映したい場合は DNS/Proxy 同期が動作すること
- `fukaya-sus.lab` などをローカルで引かせる場合は、内蔵 DNS サーバーが `127.0.0.1:53` で待ち受けられること

## 3. ダッシュボード全体の構成

ダッシュボードは SPA ですが、URL ルーティングではなくタブ切り替えで MPA 風に操作します。

主要画面:
1. ホーム
2. アプリ一覧
3. アプリ登録
4. アプリ詳細

### 3.1 画面上部

ヘッダーには次の要素があります。

- タイトル: `配信基盤ダッシュボード`
- 実行モード表示: `dry-run` または `execute`
- `DNS/Proxy 同期` ボタン
- `最新状態に更新` ボタン

補足:
- データは手動更新だけでなく自動更新されます
- そのため通常は手動更新を押さなくても状態は追従します
- ホーム画面に DNS サーバーの待受状態が表示されます

### 3.2 自動更新

現在の更新仕様:
- システム状態、アプリ一覧、イベント一覧: `15秒ごと` に自動更新
- ブラウザへフォーカスが戻った時: 自動更新
- アプリ詳細のログを開いている間: `5秒ごと` にログ自動更新

### 3.3 通知表示

成功・失敗メッセージは、ボディ領域を圧迫しないトースト表示です。

例:
- `アプリを登録しました。`
- `ログ取得に失敗しました。`

## 4. ホームタブ

ホームは、現在の全体状況を素早く把握するための画面です。

### 4.1 ステータスカード

上部のカードで次を確認できます。

- `登録アプリ`
- `稼働中`
- `不安定`
- `失敗`

用途:
- 今、全体が健全かどうかを一目で確認する
- 運用開始時に異常件数を先に見る

### 4.2 クイックメニュー

中央のクイックメニューから次の画面へ移動できます。

- `アプリ一覧へ`
- `アプリ登録へ`

### 4.3 注意が必要なアプリ

このパネルには、状態が `Failed` または `Degraded` のアプリが表示されます。

表示内容:
- アプリ名
- ホスト名
- 最新エラータイトル
- 状態バッジ
- `詳細へ` ボタン

ポイント:
- パネル内スクロール対応です
- 件数が多くてもホーム全体を崩さずに確認できます

### 4.4 最近のイベント

最近発生したイベントが新しい順に表示されます。

表示内容:
- イベントタイトル
- 対象アプリ名
- メッセージ
- 発生時刻

長いメッセージの扱い:
- 改行を含む
- ある程度以上長い

上記の場合は `詳細を開く` で展開できます。

この欄で見える内容の例:
- アプリ登録完了
- 配備開始
- リポジトリ取得完了
- `docker compose` 実行開始
- DNS/Proxy 同期開始
- 配備失敗時の詳細エラー

## 5. アプリ一覧タブ

アプリ一覧は、登録済みアプリの横断確認に使います。

### 5.1 表示項目

各アプリについて以下を表示します。

- アプリ名
- モード (`Standard` / `Headless`)
- 現在コミット
- 状態
- 最新ジョブ種別とジョブ状態
- 最新ジョブ進捗メッセージ
- 最新エラータイトル
- 最新エラーメッセージ
- 公開先ホスト名
- 更新有無
- 最終更新日時
- `詳細へ` ボタン

### 5.2 この画面で分かること

以前は「失敗した事実」は分かっても「今どこで詰まっているか」が見えにくい状態でした。現在は次も一覧から読めます。

- いま deploy ジョブが `queued` なのか `running` なのか
- 現在の進捗メッセージ
- 直近の失敗内容の要約

進捗メッセージの例:
- `GitHub リポジトリを取得しています。`
- `docker compose を起動しています。compose=docker-compose.yml`
- `公開ルートとインフラ設定を同期しています。`

### 5.3 一覧からできること

この画面では詳細確認を優先し、破壊的な操作は集約していません。

できる操作:
- `詳細へ` を押してアプリ詳細タブへ移動

## 6. アプリ登録タブ

アプリ登録は、上から順に進めるウィザード形式です。
前のステップが完了しないと後続ステップは開きません。

登録フロー:
1. GitHub URL を入力
2. ブランチを確認
3. composeファイル候補を選択
4. composeファイルを解析してサービスを選ぶ
5. アプリ情報を入力して登録

## 6.1 STEP 1: GitHub URL を入力

入力欄:
- `GitHub URL`

対応形式:
- `https://github.com/<owner>/<repo>/tree/<branch>`
- `https://github.com/<owner>/<repo>`
- `https://github.com/<owner>/<repo>.git`

実行方法:
- 入力欄からフォーカスを外す
- または `URL解析` ボタンを押す

現在の制約:
- GitHub 以外の URL には対応していません

## 6.2 STEP 2: ブランチを確認

URL 解析に成功すると、正規化されたリポジトリ URL とブランチ欄が表示されます。

表示内容:
- 正規化URL
- ブランチ入力欄
- 自動取得された branch 候補
- 取得ファイル数
- 必要に応じた警告

ブランチ解決ルール:
- `/tree/<branch>` URL の場合:
  - GitHub のブランチ一覧を取得
  - 最長一致で branch を解決
- `/repo` または `.git` URL の場合:
  - `main` 固定
  - UI 上で branch 入力は無効化

## 6.3 STEP 3: composeファイル候補を選択

このステップでは、GitHub 上のファイル一覧を元に compose 系ファイル候補を提示します。

表示内容:
- `composeファイル候補`
- `その他の YAML ファイル`
- compose ファイル手動入力欄

選び方:
1. 自動検出された composeファイル候補をクリック
2. 見つからない場合は他の YAML をクリック
3. さらに必要なら compose パスを手入力して `compose解析`

候補の元データ:
- GitHub API で取得したリポジトリ内ファイル一覧
- `docker-compose.yml`
- `docker-compose.yaml`
- `compose.yml`
- `compose.yaml`
- `compose.dev.yml` のような compose 名を含む YAML

## 6.4 STEP 4: composeファイルを解析してサービスを選ぶ

選択した compose ファイルは GitHub から内容を取得し、YAML として解析されます。

ここで行うこと:
- コンテナ候補となる `services` を一覧表示
- `ports` と `expose` から公開候補ポートを推定
- 公開対象の Web サービスを選択

各サービスカードに出る情報:
- サービス名
- `公開候補` かどうか
- `推定ポート`
- `ports/expose` で見つかった候補
- 推定理由

ポート検出の対象:
- `ports:`
- `expose:`
- 文字列表記
- 短縮表記
- 一部の map 形式

例:

```yaml
web:
  expose:
    - "80"
```

上のような場合、`80` を公開候補ポートとして認識します。

解析できない場合:
- サービス候補が出ないことがあります
- その場合でも次のステップで `公開サービス名` と `公開ポート` を手入力して登録できます

## 6.5 STEP 5: アプリ情報を入力して登録

入力欄:
- `アプリ名`
- `説明`
- `サブドメイン`
- `公開サービス名`
- `公開ポート`
- `モード`
- `再ビルド時データ保持`
- `デバイス要件 (カンマ区切り)`

各項目の意味:

### アプリ名
- Lab-Core 内での識別名です
- runtime 配下の作業ディレクトリ名にも使われます

### 説明
- 運用メモ用途です
- 任意入力です

### サブドメイン
- 公開ホスト名です
- プレースホルダーは現在の `rootDomain` に追従します（例: `app.lab.localhost`）
- ただし別ドメインの hostname も登録可能です。ローカル開発では、その名前を DNS または hosts で解決できるようにしてください

### 公開サービス名
- compose の `services.<name>` に対応します
- STEP 4 でサービスカードをクリックすると自動入力されます

### 公開ポート
- サービスが listen するコンテナ内ポートです
- STEP 4 の解析結果から自動入力されることがあります
- 必要に応じて手動修正できます

### モード
- `Standard`
- `Headless`

### 再ビルド時データ保持
- `true`: ボリュームを保持したまま再ビルド
- `false`: データを落として再ビルド

### デバイス要件
- デバイスマウントが必要なアプリ向けの追加入力です
- 例: `/dev/bus/usb, /dev/ttyUSB0`

### 登録後の挙動

`登録して配備キューに追加` を押すと:
1. アプリ構成を DB に登録
2. 初回 deploy ジョブを作成
3. 自動でアプリ詳細タブへ移動
4. 詳細画面で進行状況を追える

## 7. アプリ詳細タブ

アプリ詳細は、運用操作をまとめて行う画面です。

## 7.1 概要カード

表示内容:
- アプリ名
- 状態
- 公開先 URL
- default branch
- current commit
- previous commit

`一覧へ戻る` ボタンから一覧へ戻れます。

## 7.2 進行状況カード

このカードは、現在または直近のジョブ進捗を表示します。

表示内容:
- ジョブ種別
- ジョブ状態 (`queued` / `running` / `succeeded` / `failed`)
- 最新進捗メッセージ
- 作成時刻
- 開始時刻
- 終了時刻

用途:
- 配備がどこまで進んだかを見る
- 今ちゃんと進んでいるのかを確認する
- 失敗した場合に、どの段階まで到達したかを把握する

## 7.3 最新エラーカード

直近のエラーイベントがある場合に表示されます。

表示内容:
- エラータイトル
- 発生時刻
- エラー本文

一覧画面よりも長い内容を確認できます。

## 7.4 運用アクション

実行できる操作:
- `再起動`
- `再ビルド`
- `更新確認`
- `更新適用`
- `ロールバック`
- `ログを開く` / `ログを閉じる`

各操作の意味:

### 再起動
- `docker compose restart` 相当
- コンテナを再起動したい時に使います

### 再ビルド
- 現在の UI ではデータ保持あり (`keepData=true`) で実行します
- コード変更やイメージ再構築が必要な時に使います

### 更新確認
- remote 側の commit と比較して更新有無を反映します

### 更新適用
- 最新 remote commit を取得して再配備します

### ロールバック
- `previous_commit` が存在する場合のみ押せます
- 1世代前の commit に戻します

## 7.5 最近の進行イベント

対象アプリに紐づく最近のイベントだけを表示します。

ここで追える内容:
- 配備開始
- リポジトリ取得開始/完了
- `docker compose` 起動開始
- 公開設定同期開始
- 配備成功
- 配備失敗

長い内容は `詳細を開く` で展開できます。

## 7.6 ログ確認

`ログを開く` を押すとログパネルが開きます。

操作項目:
- サービス選択
- 表示行数 (`100 / 200 / 500 / 1000`)
- 自動スクロール ON/OFF
- `ログ更新`

現在の挙動:
- ログパネルを開いている間は `5秒ごと` に自動更新
- 手動で `ログ更新` も可能

`dry-run` 時:
- 擬似ログやイベント由来の内容が表示されます

`execute` 時:
- `docker compose logs` の実ログが表示されます

### サービス選択の考え方
- `全サービス`: まず全体を見る時に使う
- `web` や `api`: 特定コンテナだけ追いたい時に使う

### 配備失敗の見方
配備中に異常が出た場合は、次の順に見ると追いやすいです。

1. 進行状況カード
2. 最新エラーカード
3. 最近の進行イベント
4. ログ確認

## 7.7 削除

削除パネルでは、削除モードの選択と確認用アプリ名入力が必要です。

手順:
1. 削除モードを選ぶ
2. 確認用アプリ名に対象アプリ名を正確に入力
3. `削除ジョブを開始`

削除モード:
- `config_only`
  - DB 上の構成削除が中心
- `source_and_config`
  - 構成 + runtime 上のソース削除
- `full`
  - 構成 + ソース + データ削除

注意:
- 破壊的操作です
- アプリ名が一致しないと開始できません

## 8. よく使う運用フロー

## 8.1 新しい GitHub アプリを登録する
1. アプリ登録タブを開く
2. GitHub URL を入力して `URL解析`
3. branch を確認
4. composeファイル候補を選ぶ
5. サービス候補から公開対象を選ぶ
6. アプリ名、サブドメインなどを入力
7. 登録
8. 自動で開くアプリ詳細で進捗を見る

## 8.2 デプロイが進んでいるか確認する
1. アプリ一覧で対象アプリの最新ジョブ欄を見る
2. `詳細へ`
3. 進行状況カードと最近の進行イベントを見る
4. 必要ならログを開く

## 8.3 デプロイ失敗時に原因を調べる
1. アプリ一覧で失敗状態を確認
2. 詳細へ移動
3. 最新エラーカードを確認
4. 最近の進行イベントを展開
5. ログを開いて対象サービスを絞る

## 8.4 アプリを更新する
1. 詳細画面で `更新確認`
2. 更新ありなら `更新適用`
3. 問題が出たら `ロールバック`

## 8.5 アプリを削除する
1. 詳細画面の削除パネルへ進む
2. 削除モードを選ぶ
3. 確認用アプリ名を入力
4. 削除ジョブを開始
5. 一覧から消えたことを確認

## 9. 状態表示の意味

### 9.1 アプリ状態
- `Draft`: 下書き状態
- `Build Pending`: 初回 deploy 待ち
- `Cloning`: GitHub リポジトリ取得中
- `Deploying`: compose 起動中
- `Running`: 稼働中
- `Degraded`: 稼働中だが不安定
- `Stopped`: 停止
- `Failed`: 失敗
- `Rebuilding`: 再ビルド中
- `Deleting`: 削除中

### 9.2 ジョブ状態
- `queued`: キュー待ち
- `running`: 実行中
- `succeeded`: 成功
- `failed`: 失敗

## 10. イベントとログの使い分け

### イベントを見るべき場面
- 操作が始まったか確認したい
- どの段階まで進んだか見たい
- deploy / update / delete の大きな流れを追いたい

### ログを見るべき場面
- コンテナ内部で何が起きたか見たい
- Web サーバーや API の起動失敗を確認したい
- イベントだけでは足りない詳細を見たい

## 11. 現在の制約と注意点

- GitHub 以外のソース URL には対応していません
- `.git` URL や `/repo` URL は `main` 固定です
- 画面遷移は URL ではなくタブ状態で管理しています
- compose 解析はヒューリスティックです
- `ports` / `expose` からかなりのケースを拾いますが、すべての YAML 書式を完全網羅しているわけではありません
- compose 解析に失敗しても、公開サービス名と公開ポートを手入力して登録できます

## 12. 確認に使う代表コマンド

- 全体ビルド: `yarn build`
- backend のみ: `yarn workspace @lab-core/backend build`
- dashboard 型チェック: `yarn workspace @lab-core/dashboard exec tsc --noEmit`
- dashboard build: `yarn workspace @lab-core/dashboard build`
